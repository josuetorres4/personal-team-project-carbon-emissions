# Checkpoint 3 — Implementation Documentation

## What We Built

sust-AI-naible is a closed-loop multi-agent system that optimizes cloud workloads for carbon emissions. This checkpoint moved the system from a fully simulated demo into a live, deployed application with real LLM reasoning and real grid data.

---

## 1. Groq LLM Integration

**What changed:** The system already had Groq support in the codebase (`src/agents/base.py:77–95`). We activated it by creating a `.env` file with a live API key.

**How it works:**
- `LLMProvider("auto")` checks for `GROQ_API_KEY` in the environment first, then `OPENAI_API_KEY`, then falls back to mock
- Uses the OpenAI Python client pointed at `https://api.groq.com/openai/v1` with model `llama-3.3-70b-versatile`
- Rate-limit protection: exponential backoff (5 retries), 300s final wait, then deterministic fallback
- Token budget: capped at `MAX_TOTAL_LLM_TOKENS` per pipeline run to avoid hitting Groq's daily limit

**Rate limit strategy for this checkpoint:**
- `MAX_LLM_RATIONALES=5` — only the top 5 recommendations (by carbon impact) get live LLM rationales
- `MAX_LLM_TICKETS=5` — only 5 Jira ticket bodies generated by Groq
- `MAX_LLM_RISK_ASSESSMENTS=5` — only 5 governance risk assessments via Groq
- All others fall back to deterministic template responses

**Pipeline run result:**
```

Agent reasoning steps:
  planner:    9 reasoning steps
  governance: 5 reasoning steps
  executor:   7 reasoning steps
  copilot:   11 reasoning steps
```

## 2. Real Carbon Intensity Data

**File:** `src/data/carbon_intensity_real.py`

Three data sources combined, one per geographic region group:

| Region | AWS Name | Data Source | Method |
|--------|----------|-------------|--------|
| us-east-1 | Virginia | **EIA Open Data API** | Hourly fuel mix → gCO₂/kWh |
| us-west-2 | Oregon | **EIA Open Data API** | Hourly fuel mix → gCO₂/kWh |
| eu-west-1 | Ireland | ENTSO-E (no token yet) → synthetic fallback | — |
| eu-north-1 | Stockholm | ENTSO-E (no token yet) → synthetic fallback | — |
| ap-south-1 | Mumbai | **Ember Climate** static | Annual avg + diurnal variation |

**EIA Integration:**
- API: `https://api.eia.gov/v2/electricity/rto/fuel-type-data/data/`
- Balancing authorities: PJM (Virginia), BPAT/BPA (Oregon)
- Fetches last 1 day of data, tiles it across the simulation period to minimize API calls
- Emission factors per fuel type: coal 995, gas 410, oil 840, nuclear/hydro/wind/solar 0 gCO₂/kWh
- Caches responses in `data/.cache/` for `CARBON_DATA_CACHE_HOURS` (default 6h)

**Ember (India):**
- Static annual average: ~700 gCO₂/kWh (Ember Climate 2023)
- Adds sinusoidal diurnal variation ±50 gCO₂/kWh to avoid flat lines
- No API key required

**Fallback chain:**
1. Real API data (if keys present and request succeeds)
2. Per-region synthetic fallback (if one region's API fails)
3. Full synthetic fallback (if `USE_REAL_CARBON_DATA=false`)

**Output schema** (unchanged from synthetic — drop-in replacement):
```
timestamp | region | intensity_gco2_kwh | intensity_lower | intensity_upper | source
```

## 3. Azure VM Traces Loader

**File:** `src/data/azure_traces.py`

**Dataset:** [Azure Public Dataset VM Traces 2019](https://github.com/Azure/AzurePublicDataset)

**Download:**
```bash
wget https://azurepublicdatasettraces.blob.core.windows.net/azurepublicdatasetv2/trace_data/vmtable/vmtable.csv.gz
gunzip vmtable.csv.gz
mkdir -p data/azure_traces && mv vmtable.csv data/azure_traces/
```

**Column mapping:**

| Azure Column | Job Field | Logic |
|---|---|---|
| `vmId` | `job_id` | `f"az-{vmId}"` |
| `core` | `vcpus` | Direct |
| `vmCreated` / `vmDeleted` | `started_at`, `ended_at`, `duration_hours` | Normalized to sim window |
| `vmCategory` | `category` | Delay-insensitive→SUSTAINABLE, Interactive→URGENT, else→BALANCED |
| `vmCategory` + heuristics | `workload_type` | GPU→model_training, short interactive→ci_cd, long→production, etc. |
| `subscriptionId` hash | `team_id` | Mapped to one of 10 team names deterministically |
| `subscriptionId` hash | `region` | Weighted: 40% us-east-1, 20% us-west-2, 15% eu-west-1, 10% eu-north-1, 15% ap-south-1 |
| `core >= 16 AND avgCpu > 80` | `gpu_count = 1` | Heuristic for training workloads |

**Performance:** Loads only required columns (`usecols`), defaults to `max_jobs=30,000` sample.

**Fallback:** If CSV not found, returns empty list → orchestrator falls back to synthetic workloads.

## 4. Orchestrator Changes

**File:** `src/orchestrator.py`

Two one-line swaps in the SENSE step:

```python
# Before
jobs = generate_workloads(sim_start, num_days=sim_days, seed=seed)
intensity_df = generate_intensity_timeseries(sim_start, num_days=sim_days, seed=seed)

# After
jobs = get_workload_data(sim_start, sim_days=sim_days, seed=seed)

```

Both new functions have identical signatures and return types — fully backward-compatible.

## 9. Streamlit Cloud Deployment

**File:** `.streamlit/config.toml`

**Steps:**
1. Run pipeline locally → commit `data/` outputs to repo (pre-generated fallback)
2. Push to GitHub
3. [share.streamlit.io](https://share.streamlit.io) → New app → select `dashboard.py`
4. Add secrets in Settings → Secrets:
   ```toml
   GROQ_API_KEY = "gsk_..."
   LLM_PROVIDER = "auto"
   EIA_API_KEY = "..."
   USE_REAL_CARBON_DATA = "true"
   ```
5. App loads immediately from pre-generated data; "Re-run Pipeline" button regenerates if API keys are configured
